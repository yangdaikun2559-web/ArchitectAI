# ArchitectAI 仿真实验软件参赛定位与功能改造方案

## 一、方向结论

建议将 ArchitectAI 的参赛方向明确调整为：

**常规项目 - 课件 - 仿真实验软件**

推荐作品名称：

**ArchitectAI：物联网虚拟仿真实验与 AI 工程学习课件**

或更聚焦一点：

**ArchitectAI：基于 STM32 的物联网虚拟仿真实验课件**

这个方向比“AI 生成平台”更贴合比赛，也比普通课件更有亮点。活动指南中对课件的说明包含“仿真实验软件等均可报送”，因此你仍然报“课件”赛道，但作品定位写成“仿真实验软件型课件”。

## 二、为什么适合做仿真实验软件

ArchitectAI 当前已有这些基础：

- 能生成物联网实验项目方案。
- 能生成 CAD 接线图。
- 能展示 BOM 元器件清单。
- 能进行硬件引脚分配。
- 能做电气安全/兼容性检测。
- 能生成代码并导出工程。
- 已有学习任务、班级提交和教师反馈能力。

这些能力天然可以组织成一个“虚拟实验流程”：

学生不只是看课件，而是在软件中完成一次虚拟物联网实验。

## 三、参赛叙事应如何调整

不要主打：

> AI 物联网智能生成平台

建议主打：

> 本作品是一款面向物联网与单片机课程的虚拟仿真实验软件。学生在课件中完成从实验需求、元器件选型、虚拟接线、安全检测、代码理解到工程导出的完整实验流程。系统通过 AI 提供实验脚手架，通过仿真接线和安全检测降低硬件损坏风险，通过教师端评价支持实验过程性评价。

这样评委会更容易判断它属于“课件/仿真实验软件”，而不是一个泛技术工具。

## 四、核心实验场景建议

主实验建议固定为：

**实验名称：基于 STM32 的智能温室环境监测与报警系统虚拟实验**

实验目标：

1. 理解物联网系统由主控、传感器、显示模块、报警模块和通信模块组成。
2. 掌握 STM32 常见接口类型，包括 GPIO、ADC、I2C、USART。
3. 能根据虚拟接线图判断各模块的连接关系。
4. 能识别 5V/3.3V 电平风险、共地要求和执行器驱动风险。
5. 能对照 `pins.h` 与 `main.c` 理解硬件连接和程序逻辑。
6. 能导出工程文件，为真实硬件实验做准备。

硬件模块：

- STM32F103C8T6 Blue Pill
- DHT22 温湿度传感器
- MQ-2 气体/烟雾传感器
- OLED SSD1306 显示屏
- 有源蜂鸣器
- ESP8266 WiFi 模块

## 五、功能应如何改成“仿真实验软件”

### P0-1：增加“虚拟仿真实验”主入口

首页增加入口：

**开始虚拟仿真实验：STM32 智能温室**

点击后进入实验，而不是普通项目生成。

入口下方写清楚实验流程：

需求分析 -> 器件选型 -> 虚拟接线 -> 安全检测 -> 代码探究 -> 工程导出 -> 实验评价

### P0-2：新增“实验任务面板”

左侧或顶部显示实验步骤：

1. 实验情境
2. 实验目标
3. 器件认识
4. 虚拟接线
5. 安全检测
6. 程序探究
7. 实验报告
8. 教师评价

每一步都要有学生要完成的任务，而不是只有软件结果。

### P0-3：把 CAD 接线图升级为“虚拟接线实验区”

现有 CAD 图可以作为基础，但文案和交互要改成实验表达：

- 显示“当前实验接线”。
- 显示每根线的信号类型。
- 点击元器件查看引脚说明。
- 点击引脚查看接线原因。
- 修改接线后触发安全检测。
- 展示“实验结论：接线是否通过”。

如果短期来不及做复杂拖拽，也可以先做“下拉选择引脚 + 实时检测”的轻量仿真。

### P0-4：强化“电气安全仿真检测”

这是仿真实验软件的核心亮点，应独立成一个模块：

**电气安全仿真检测报告**

检测项建议：

- 引脚冲突：同一 GPIO 是否被多个非总线设备占用。
- 总线规则：I2C SDA/SCL 是否正确共享。
- 电平风险：5V 模块是否直接接入 3.3V MCU 信号引脚。
- 执行器驱动：蜂鸣器、继电器、电机是否需要驱动电路。
- 共地检查：所有外设是否需要与主控共地。
- STM32 接口适配：ADC、I2C、USART、GPIO 是否匹配。

显示方式建议：

- 通过：绿色。
- 警告：黄色。
- 错误：红色。
- 每条检测结果都给出“为什么”和“如何修改”。

### P0-5：增加“实验数据仿真”

为了更像仿真实验软件，建议增加一个小型数据模拟区：

- 温度滑块。
- 湿度滑块。
- 气体浓度滑块。
- OLED 屏幕模拟显示。
- 蜂鸣器状态模拟：正常/报警。
- 阈值设置：温度阈值、气体阈值。

学生调节数据后，能看到：

- OLED 显示如何变化。
- 报警逻辑是否触发。
- 代码中的阈值判断对应哪一段。

这是最能把“接线 + 代码 + 实验现象”连起来的功能。

### P0-6：增加“实验报告生成/提交”

学生完成实验后，生成一份实验报告草稿：

- 实验名称。
- 实验目标。
- 器件清单。
- 接线说明。
- 安全检测结果。
- 程序逻辑说明。
- 实验现象记录。
- 问题与改进。

可以先不做复杂导出，先在页面中展示并允许复制/提交到班级。

## 六、页面结构建议

建议把参赛演示模式做成一个完整页面：

```text
虚拟仿真实验：STM32 智能温室

左侧：实验步骤导航
中间：当前实验操作区
右侧：实验记录/安全检测/教师提示
底部：上一环节、下一环节、提交实验报告
```

每个步骤的核心内容：

| 步骤 | 页面重点 |
|---|---|
| 实验情境 | 温室监测问题、实验任务、学习目标 |
| 器件认识 | STM32、DHT22、MQ-2、OLED、蜂鸣器、ESP8266 |
| 虚拟接线 | CAD 图、引脚表、信号类型 |
| 安全检测 | 规则检测、风险解释、修改建议 |
| 数据仿真 | 传感器数据滑块、OLED 模拟、蜂鸣器状态 |
| 程序探究 | `pins.h`、采集逻辑、阈值判断、报警控制 |
| 实验报告 | 自动汇总实验记录 |
| 教师评价 | 提交到班级、教师反馈 |

ArchitectAI 是一款面向物联网与单片机课程的虚拟仿真实验软件。作品以“基于 STM32 的智能温室环境监测与报警系统”为主实验，引导学生完成需求分析、器件选型、虚拟接线、电气安全检测、传感器数据仿真、代码探究和实验报告提交。课件集成 AI 实验脚手架、CAD 接线图、元器件百科、BOM 清单、5V/3.3V 电平风险提示、引脚冲突检测、OLED 显示与蜂鸣器报警模拟等功能，帮助学生在真实硬件操作前理解系统结构和安全规范。教师可通过学习任务与提交记录开展过程性评价。作品将传统物联网实验中抽象、易错、易损的环节转化为可视化、可交互、可反馈的虚拟实验过程，有助于提升学生工程思维、实践能力和数字素养。
