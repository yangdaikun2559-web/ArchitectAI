# ArchitectAI 课件赛道资格评审与设计优化报告

## 一、评审依据

本报告依据《四川省教育厅关于举办2026年四川省教师数字素养提升实践活动的通知》及《2026年四川省教师数字素养提升实践活动指南（常规项目）》进行评审。

常规项目指南中，“课件”是指基于数字化、网络化、智能化信息技术，围绕教学内容、学情、目标、过程、方法与评价进行整体设计并制作完成的应用软件。各类教学软件、学生自主学习软件、教学评价软件、仿真实验软件等均可报送。

本项目当前拟申报方向为：

**常规项目 - 课件**

建议参赛作品名称为：

**《ArchitectAI：基于 STM32 的物联网探究式学习与仿真实验课件》**

## 二、参赛资格判断

### 1. 赛道匹配结论

**结论：具备申报课件赛道资格，且课件赛道是当前项目最合适的申报方向。**

ArchitectAI 当前已具备以下课件属性：

1. 具备明确的教学任务：基于 STM32 的智能温室环境监测与报警系统。
2. 具备教学软件形态：通过网页应用完成需求输入、方案生成、接线展示、代码导出。
3. 具备仿真实验特征：通过 CAD 接线图、安全检测和虚拟工程生成辅助实验教学。
4. 具备学生自主学习特征：学生可根据自然语言需求完成方案构建、接线理解和代码阅读。
5. 具备教学评价潜力：可通过项目生成记录、安全警告、下载记录等支撑过程评价。

### 2. 硬性资格条件

以下条件必须满足：

| 条件 | 当前判断 | 说明 |
|---|---|---|
| 第一作者为教师或教育技术工作者 | 待确认 | 这是参赛资格硬门槛 |
| 每件作品作者不超过 3 人 | 可满足 | 申报时需控制作者人数 |

### 3. 不建议申报的方向

不建议报“微课”，因为本作品不是单一有声教学视频，而是应用软件型课件。

不优先报“融合创新应用教学案例”或“信息化教学课程案例”，除非后续具备完整课堂实录、真实应用数据、推广效果和课堂活动录像。

## 五、推荐设计方案

### 1. 核心设计定位

建议将作品定位为：

**面向单片机与物联网课程的 AI 辅助探究式学习课件。**

不要把它主要表述为“AI 物联网生成平台”，而应表述为“课件软件”：

> 本课件以 STM32 智能温室环境监测任务为载体，通过 AI 方案生成、虚拟接线、安全检测、代码理解和教师评价，帮助学生完成从生活需求到工程验证的探究式学习过程。

### 2. 主课例设计

主课例固定为：

**基于 STM32 的智能温室环境监测与报警系统**

硬件组合：

- STM32F103C8T6
- DHT22 温湿度传感器
- MQ-2 气体/烟雾传感器
- OLED SSD1306 显示屏
- 有源蜂鸣器
- ESP8266 WiFi 拓展模块

教学任务：

1. 学生输入智能温室需求。
2. AI 将自然语言需求转化为结构化工程方案。
3. 课件生成器件清单和 CAD 接线图。
4. 学生查看元器件百科，理解不同接口类型。
5. 课件给出安全检测报告。
6. 学生阅读 STM32 代码，理解 GPIO、I2C、ADC、USART 的作用。
7. 学生导出工程并准备实物验证。
8. 教师根据过程数据进行评价。

### 3. 教学目标设计

建议写入设计说明：

1. 能将生活化物联网需求拆解为主控、传感器、显示、报警和通信模块。
2. 能识别 STM32 常见接口，包括 GPIO、I2C、ADC 和 USART。
3. 能根据 CAD 接线图判断 DHT22、MQ-2、OLED、蜂鸣器和 ESP8266 的连接关系。
4. 能理解 `pins.h` 与硬件接线之间的对应关系。
5. 能说明阈值判断和蜂鸣器报警的程序逻辑。
6. 能形成基本的电气安全意识，如共地、5V 与 3.3V 电平保护、执行器驱动保护。

### 4. 教学流程设计

建议采用六步流程：

1. **创设情境**：温室需要实时监测温湿度和空气质量。
2. **需求建模**：学生输入自然语言需求，AI 优化为结构化方案。
3. **虚拟接线**：生成 STM32 接线图，展示各模块接口关系。
4. **安全检测**：检查共地、电平、执行器驱动、接口冲突。
5. **代码探究**：阅读 `pins.h` 和 `main.c`，理解硬件与代码映射。
6. **评价拓展**：教师查看任务完成情况，学生进行阈值修改或联网拓展。

## 六、必须优先改动的内容

### P0-1：完成 STM32 Demo 链路统一

目标：确保加载演示项目后，所有页面、接线图、代码、README、BOM 均围绕同一个 STM32 智能温室课例。

当前已完成主要改动：

- 已新增 STM32 参赛 Demo。
- 已保留 ESP32 生成路线。
- 已将 Demo 代码改为 STM32 标准外设库风格。

还需继续检查：

- 页面标题是否仍有“平台生成器”过重的工具化表达。
- 下载中心是否显示适合 STM32/Keil 的运行说明。
- BOM 是否能正确识别 STM32、DHT22、MQ-2、OLED、蜂鸣器、ESP8266。

### P0-2：增加安全检测报告展示区

这是最能对应“技术运用”和“创新实用”的亮点。

建议展示三类信息：

**检测通过**

- OLED 使用 I2C，总线分配为 PB6/PB7。
- DHT22 使用 PA0 单总线数据引脚。
- MQ-2 使用 PA1 ADC 输入。
- 蜂鸣器使用 PB0 GPIO 输出。

**安全提示**

- 所有模块必须共地。
- MQ-2 和蜂鸣器涉及 5V 工作电压。
- STM32 GPIO 为 3.3V 逻辑，需注意电平保护。
- 蜂鸣器建议使用三极管或驱动模块。

**教学提示**

- I2C、单总线、ADC、GPIO、USART 属于不同接口类型。
- 接线图与 `pins.h` 应逐项对应。

### P0-3：增加课堂演示入口

建议按钮名称：

- `课堂演示：STM32 智能温室`
- `加载 STM32 参赛课例`
- `进入课件演示模式`

该入口应直接加载 STM32 Demo，并跳转至结果页或教学流程页，便于录制视频。

### P0-4：完成演示视频

视频要求：

- MP4 格式。
- 画面尺寸 1280×720 以上。
- 一般不超过 10 分钟。
- 清晰演示课件实际运行过程。
- 配同步语音讲解。

当前已有《10分钟演示视频脚本正式稿》，下一步应根据真实页面录制。

## 七、强烈建议补充的内容

### P1-1：学生任务单

任务单建议包括：

1. 写出智能温室系统由哪些模块组成。
2. 标注 DHT22、MQ-2、OLED、蜂鸣器分别连接到 STM32 哪个引脚。
3. 判断哪些器件涉及 5V 电压，说明应如何保护 STM32。
4. 在 `pins.h` 中找到 OLED、DHT22、MQ-2、蜂鸣器对应的宏定义。
5. 修改温度阈值或气体阈值，解释报警逻辑变化。

### P1-2：课堂评价量表

建议评价维度：

| 评价维度 | 评价内容 |
|---|---|
| 需求分析 | 能否将生活需求拆成系统模块 |
| 器件选型 | 能否说明各器件作用 |
| 接线规范 | 能否识别接口类型和共地要求 |
| 安全意识 | 能否说明 5V 与 3.3V 电平风险 |
| 代码理解 | 能否解释 GPIO、I2C、ADC、阈值判断 |
| 工程拓展 | 能否提出联网或传感器拓展方案 |

### P1-3：代码教学解读

建议在代码页旁边增加教学解读：

- `GPIO_Configuration`：配置输入输出引脚。
- `I2C_Configuration`：初始化 OLED 通信总线。
- `ADC_Configuration`：配置 MQ-2 气体传感器采集通道。
- `DHT22_ReadData`：完成温湿度读取。
- `Buzzer_Set`：控制蜂鸣器报警。
- `TEMP_THRESHOLD` / `GAS_THRESHOLD`：体现阈值判断。

### P1-4：教师评价看板

建议把后台包装为教师评价视角：

- 学生是否完成需求优化。
- 是否生成接线图。
- 是否存在安全警告。
- 是否下载工程代码。
- 高频错误集中在哪类接口。
- 教师下一步讲解建议。
