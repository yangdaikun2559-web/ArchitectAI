# 物联网网页项目需求分析

**文档版本**：V1.3  
**编写日期**：2026-05-20  
**适用范围**：物联网项目需求描述、方案生成、电路图生成、代码生成与下载

## 1. 项目背景

本项目拟开发一个面向物联网场景的网页平台。用户可以在网页中输入自己的项目需求，例如“做一个温湿度监测系统”或“做一个智能灯控系统”，平台根据需求自动生成：

1. 项目方案说明
2. 硬件电路图或接线图
3. 对应项目代码
4. 可下载的完整代码包

该平台的目标是降低物联网项目从需求到原型实现的门槛，提高方案产出效率，适合初学者、开发者、教学演示和原型验证场景。

## 2. 项目目标

- 让用户用自然语言描述需求即可生成物联网项目方案
- 自动输出适配需求的硬件连接方案和电路图
- 自动生成可运行的项目代码
- 支持代码包一键打包下载
- 支持后续再次修改参数并重新生成结果

## 3. 目标用户

- 物联网开发初学者
- 教学和培训机构
- 创客和原型开发者
- 小型硬件项目团队

## 4. 优先支持的平台与器件

### 4.1 优先支持的硬件平台

- STM32
- ESP32

### 4.2 优先支持的传感器与外设

- DHT11
- DHT22
- LED
- 蜂鸣器
- 继电器
- 光敏传感器
- OLED 屏

### 4.3 默认支持方向

- 优先围绕 STM32 + ESP32 的常见物联网框架生成方案
- 优先输出上述器件组合的电路图、接线说明和示例代码
- 后续再逐步扩展更多传感器、通信模块和执行器

## 5. AI 提示词优化规则

本项目取消多轮追问流程。用户输入原始需求后，系统先把这段输入优化成一份更完整、可直接用于生成的提示词，再进入方案生成链路。用户可以一键采用优化结果，也可以直接修改优化后的提示词继续生成。

### 5.1 优化目标

- 将口语化、零散的需求转换成结构化提示词
- 补足场景、平台、器件、联网方式、显示方式、供电方式、输出格式等信息
- 尽量减少用户来回填写
- 保留原始输入与优化后的提示词，便于回看和二次编辑

### 5.2 优化策略

| 项目 | 优化方式 | 示例 |
|---|---|---|
| 主控平台 | 根据场景推荐，未指定时给出候选值。 | 推荐使用 ESP32 或 STM32。 |
| 传感器/外设 | 从支持器件库中匹配最贴近场景的组合。 | 温湿度场景优先 DHT11、DHT22。 |
| 联网方式 | 若涉及远程查看或云端控制，自动补全联网方案。 | Wi-Fi + MQTT 或 HTTP。 |
| 显示方式 | 若需要本地展示数据，补全显示模块。 | OLED 屏。 |
| 供电方式 | 按原型项目默认补全为最易落地的供电方式。 | USB 5V。 |
| 控制方式 | 根据场景自动补全自动控制、本地控制或远程控制。 | 远程 App/网页控制。 |
| 输出范围 | 补全用户希望得到的产出类型。 | 方案、电路图、代码包、README。 |

### 5.3 优化输出要求

- 系统在结果中明确标注“原始输入”“优化提示词”“推荐值”
- 系统生成的提示词应可直接进入后续生成链路
- 如果某项无法可靠推断，系统允许使用通用默认值，并在优化结果中提示可编辑
- 所有优化结果应保留到项目记录中，便于后续重新生成和调整

### 5.4 优化示例

用户输入：

> 做一个智能家居系统。

系统优化后的提示词示例：

> 请为一个基于 ESP32 的智能家居原型生成项目方案，包含 LED 灯控、蜂鸣器告警、OLED 状态显示，联网方式为 Wi-Fi + MQTT，供电方式为 USB 5V，输出内容包括硬件电路图、接线说明、示例代码、README 和可下载代码包。

## 6. 页面说明

平台应至少包含以下主要页面。

| 页面 | 作用 | 主要内容 | 关键操作 |
|---|---|---|---|
| 需求输入页 | 用户首次提交项目需求。 | 需求文本框、示例提示、支持范围提示、提交按钮。 | 输入需求并创建任务。 |
| 提示词优化页 | 展示系统优化后的提示词，供用户快速核对和编辑。 | 原始输入、优化提示词、推荐值、快捷标签、编辑入口。 | 一键采用优化结果或手动修改后继续生成。 |
| 生成进度页 | 展示生成任务状态。 | 任务状态、进度条、步骤日志、预计耗时。 | 查看生成中、失败、完成状态。 |
| 结果页 | 展示生成后的整体结果。 | 方案摘要、器件清单、电路图入口、代码入口、下载入口。 | 查看完整结果并跳转预览。 |
| 电路图预览页 | 预览硬件接线图。 | 图纸预览、引脚说明、放大缩小、导出按钮。 | 查看和导出电路图。 |
| 代码预览页 | 预览生成的代码内容。 | 文件树、代码高亮、文件切换、复制按钮。 | 在线查看代码结构和文件内容。 |
| 下载页 | 下载生成的成果包。 | 文件列表、版本信息、压缩包下载按钮、下载记录。 | 下载 ZIP 包和图纸文件。 |
| 历史记录页 | 查看历史项目。 | 项目列表、搜索、筛选、重新生成入口。 | 查看、复制、重跑历史项目。 |
| 登录页 | 用户登录和注册。 | 手机号/邮箱、密码、验证码、找回入口。 | 登录、注册、找回密码。 |
| 后台管理页 | 管理模板、规则、日志和记录。 | 主控板、传感器、模板、规则、日志、统计、权限。 | 新增、编辑、启停、审核。 |

## 7. 角色与权限

平台应区分以下角色，并严格控制访问范围。

| 角色 | 可访问范围 | 主要权限 | 限制 |
|---|---|---|---|
| 游客 | 首页、示例、需求输入页的临时体验内容。 | 可输入需求并查看临时预览结果。 | 不能保存历史、不能下载正式文件、不能访问后台。 |
| 登录用户 | 仅自己的项目、历史记录、下载页。 | 创建项目、查看结果、下载自己的代码包、重生成自己的项目。 | 不能查看他人项目，不能访问后台。 |
| 管理员 | 自己的项目 + 后台管理页。 | 维护主控板、传感器、器件库、接线规则、代码模板、README 模板、生成规则、失败日志、下载记录。 | 不能修改超级管理员权限和系统级安全配置。 |
| 超级管理员 | 全部资源与系统设置。 | 管理角色、账号、权限策略、全局配置、系统级日志和审计。 | 仅超级管理员可做高权限系统操作。 |

### 权限原则

- 用户只能访问自己的项目数据
- 下载需要鉴权，不能直接公开文件地址
- 后台管理页仅管理员及以上可见
- 超级管理员可以管理角色和全局配置

## 8. 生成结果标准

### 8.1 方案应包含什么

方案页应至少包含以下内容：

- 项目名称和应用场景
- 需求摘要和功能目标
- 原始输入和优化后的提示词
- 主控平台选择结果
- 器件清单和推荐型号
- 模块之间的关系说明
- 联网方式和通信协议
- 供电方式和电压说明
- 关键引脚或接口分配原则
- 已确认项、推荐项和优化项
- 风险提示和限制说明

### 8.2 电路图应包含什么

电路图不能只画抽象模块框图，至少要包含：

- 具体主控板型号
- 具体器件名称
- 每个器件对应的引脚连接
- 电源连接方式和电压标注
- 接口类型标注，例如 GPIO、I2C、UART、ADC、PWM
- 传感器和执行器的方向说明
- 必要时标注上拉、限流、电平转换等说明
- 可导出为 PNG、SVG 或 PDF

### 8.3 代码包应包含什么

代码包至少包含以下内容：

- 主程序
- 依赖说明
- 接线说明
- 运行步骤
- README

建议同时包含：

- 项目目录结构
- 配置文件
- 设备驱动文件
- 示例数据或测试说明
- 构建说明或烧录说明

### 8.4 README 应包含什么

README 至少应包含：

- 项目简介
- 适用场景
- 硬件清单
- 接线说明
- 依赖环境
- 运行步骤
- 烧录或部署步骤
- 常见问题和排查方法
- 安全提示和接线免责声明

## 9. 核心数据对象

平台建议围绕以下核心数据对象进行设计。

| 数据对象 | 用途 | 核心字段 | 备注 |
|---|---|---|---|
| 用户 | 保存登录和权限信息。 | 用户ID、昵称、邮箱/手机号、角色、状态、创建时间。 | 角色包括游客、登录用户、管理员、超级管理员。 |
| 项目 | 表示一个完整的物联网生成任务。 | 项目ID、用户ID、项目名称、场景类型、状态、版本号。 | 一个用户可有多个项目。 |
| 需求 | 保存用户原始输入与结构化结果。 | 需求ID、项目ID、原始文本、结构化JSON、优化提示词、标签。 | 便于重生成和审计。 |
| 提示词 | 保存原始输入和优化后的生成提示词。 | 提示词ID、项目ID、原始输入、优化提示词、编辑记录、来源、版本号。 | 用于提示词迭代和审计。 |
| 生成任务 | 跟踪生成流程。 | 任务ID、项目ID、任务类型、进度、状态、开始时间、结束时间、错误信息。 | 支持异步和超时处理。 |
| 电路图 | 保存图纸和引脚关系。 | 电路图ID、项目ID、文件地址、预览地址、引脚映射、版本号。 | 可关联多个版本。 |
| 代码包 | 保存生成后的压缩包。 | 代码包ID、项目ID、文件地址、压缩格式、校验值、生成时间。 | 下载前需鉴权。 |
| 硬件模板 | 定义主控板模板。 | 模板ID、平台、型号、默认引脚、接口能力、支持状态。 | 管理员可维护。 |
| 代码模板 | 定义代码生成规则。 | 模板ID、平台、场景、入口文件、占位符、语言。 | 可按 STM32/ESP32 区分。 |
| README 模板 | 定义文档输出内容。 | 模板ID、适用场景、章节结构、变量占位符。 | 统一文档风格。 |
| 器件库 | 定义元器件资料。 | 器件ID、类别、型号、接口类型、电压、说明。 | 如 DHT11、DHT22、LED 等。 |
| 接线规则 | 定义器件与主控的连接约束。 | 规则ID、平台、器件、引脚映射、注意事项。 | 用于自动生成接线图。 |
| 生成规则 | 定义生成逻辑和默认策略。 | 规则ID、触发条件、输出范围、默认值、限制项。 | 便于后台调整生成行为。 |
| 失败日志 | 记录生成和打包失败信息。 | 日志ID、任务ID、错误类型、错误详情、堆栈、时间。 | 用于排查问题。 |
| 下载记录 | 记录下载行为。 | 记录ID、用户ID、项目ID、文件ID、时间、IP、结果。 | 用于审计和统计。 |

## 10. 异常场景与处理

| 异常场景 | 触发条件 | 提示文案示例 | 处理方式 |
|---|---|---|---|
| 输入过短 | 用户只输入几个字，无法判断需求。 | 内容过短，已为你生成基础提示词草案，你可以直接继续编辑。 | 自动生成基础提示词并进入优化页。 |
| 需求超出支持范围 | 用户要求的主控或器件不在当前支持范围内。 | 当前优先支持 STM32、ESP32 及常用器件，已为你切换到可支持的替代方案。 | 生成兼容提示词，并使用替代模板继续。 |
| 缺少参数 | 主控、传感器、联网方式等关键项未填写。 | 已根据当前输入补全提示词，你可以直接继续生成或修改细节。 | 使用 AI 优化提示词后继续生成。 |
| 提示词优化失败 | AI 无法可靠生成优化后的提示词。 | 提示词优化失败，已回退到基础模板。 | 直接使用原始输入和基础模板继续生成。 |
| 生成失败 | 方案、电路图或代码生成过程报错。 | 生成失败，已记录日志，请稍后重试。 | 任务状态标记为失败，保留已生成内容和日志。 |
| 打包失败 | 代码已生成，但压缩打包步骤失败。 | 代码生成成功，但打包失败，可以重新打包。 | 保留代码成果，支持单独重试打包。 |
| 下载失效 | 下载链接过期或文件已失效。 | 下载链接已失效，请重新生成下载链接。 | 重新签发下载地址，重新鉴权。 |
| 未登录 | 游客访问下载、历史或自己的项目。 | 请先登录后再查看历史记录或下载文件。 | 跳转登录页或弹出登录窗口。 |
| 越权访问 | 用户访问其他人的项目或文件。 | 无权访问该项目。 | 返回 403 或提示无权限，并记录审计日志。 |
| 任务超时 | 生成耗时过长。 | 任务仍在后台处理中，请稍后在历史记录中查看结果。 | 任务继续异步执行，前端轮询或后续通知。 |

## 11. 后台管理范围

后台应支持维护以下内容。

| 管理模块 | 维护内容 | 可执行操作 |
|---|---|---|
| 主控板管理 | STM32、ESP32 等平台信息，默认引脚、接口能力、供电要求。 | 新增、编辑、启用、停用、删除。 |
| 传感器管理 | DHT11、DHT22、光敏传感器等器件信息。 | 新增、编辑、启用、停用、删除。 |
| 器件库管理 | LED、蜂鸣器、继电器、OLED 屏等通用器件。 | 新增、编辑、启用、停用、删除。 |
| 接线规则管理 | 器件与主控板的接线关系、接口类型、注意事项。 | 新增、编辑、版本更新、停用。 |
| 代码模板管理 | 不同平台和场景下的代码生成模板。 | 新增、编辑、预览、启用、停用。 |
| README 模板管理 | 文档章节结构、说明内容、变量占位符。 | 新增、编辑、预览、启用、停用。 |
| 生成规则管理 | 默认值、约束条件、输出内容限制、兼容策略。 | 新增、编辑、启用、停用。 |
| 失败日志管理 | 生成失败、打包失败、下载失败的日志记录。 | 查看、筛选、导出、备注处理结果。 |
| 下载记录管理 | 文件下载时间、用户、项目、结果等记录。 | 查看、筛选、导出、审计。 |

后台只允许管理员及以上角色访问，超级管理员可额外管理用户、角色、权限和全局策略。

## 12. 安全说明

- 输入过滤：需求文本和自动补充内容都要做过滤，防止脚本注入、HTML 注入和异常字符污染
- 文件下载鉴权：下载链接应为短期有效，并绑定用户身份和项目 ID
- 用户数据隔离：用户只能读取自己的项目、需求、任务、电路图和代码包
- 生成内容安全检查：生成后的代码、链接和文档应做基础安全扫描，避免输出明显危险或可疑内容
- 日志脱敏：失败日志和下载记录中不应泄露用户敏感信息
- 接线免责声明：电路图和接线说明仅供参考，用户在实际接线前必须自行核对官方资料和实物连接，断电后再操作

## 13. 功能需求分析表

| 序号 | 功能模块 | 需求说明 | 优先级 | 验收标准 |
|---|---|---|---|---|
| 1 | 需求提交 | 用户可通过文本框输入自己的物联网项目需求，例如“做一个温湿度监测系统”。 | 必须 | 能提交需求并创建任务。 |
| 2 | 需求解析 | 系统自动识别项目类型、功能目标、传感器/执行器、通信方式、供电方式等关键信息。 | 必须 | 能输出结构化需求摘要。 |
| 3 | 提示词优化 | 系统根据原始输入生成可直接用于生成的优化提示词。 | 必须 | 能输出原始输入、优化提示词和推荐值，并可继续编辑。 |
| 4 | 方案生成 | 系统根据需求生成整体项目方案。 | 必须 | 能生成包含主控、器件、流程、约束的方案页。 |
| 5 | 硬件电路图生成 | 根据方案自动生成硬件电路图或接线图，支持在线预览。 | 必须 | 电路图包含具体引脚和接口说明。 |
| 6 | 电路图导出 | 支持将电路图导出为 PNG、SVG、PDF 等格式。 | 必须 | 可成功下载对应文件。 |
| 7 | 代码生成 | 根据需求和硬件方案自动生成项目代码。 | 必须 | 能生成完整的代码目录结构和可预览代码。 |
| 8 | 代码包下载 | 将生成的代码、说明文档、依赖文件统一打包成压缩包供用户下载。 | 必须 | ZIP 包可下载并解压，内容完整。 |
| 9 | 项目说明文档 | 自动生成 README，说明项目用途、接线方式、运行步骤、注意事项。 | 重要 | README 内容完整且可读。 |
| 10 | 历史记录 | 保存用户历次生成的项目记录。 | 重要 | 用户可查看自己的历史项目。 |
| 11 | 再次生成 | 用户可基于历史项目修改部分参数后重新生成电路图和代码。 | 重要 | 修改后结果能同步更新。 |
| 12 | 用户登录/注册 | 支持账号体系，保存个人项目记录和下载权限。 | 重要 | 登录后可查看个人历史数据。 |
| 13 | 后台管理 | 管理员可维护硬件模板、代码模板、器件库和生成规则。 | 重要 | 后台可维护主控板、传感器、接线规则、README 模板、失败日志和下载记录。 |
| 14 | 异常提示 | 生成失败、下载失败、参数缺失时给出明确提示，并支持重试。 | 必须 | 错误信息清晰可理解。 |
| 15 | 权限控制 | 不同用户仅能访问自己的项目和下载内容，后台功能仅管理员可见。 | 必须 | 无越权访问问题。 |
| 16 | 多端适配 | 页面支持 PC 端和移动端浏览。 | 重要 | 主流浏览器下可正常使用。 |
| 17 | 统计分析 | 统计用户提交量、生成成功率、常用项目类型等数据。 | 可选 | 后台能查看基础统计数据。 |
| 18 | 计费扩展 | 后续可扩展按次数、按套餐或订阅模式收费。 | 可选 | 可接入付费逻辑。 |
| 19 | 日志与记录管理 | 管理员可查看失败日志、下载记录和操作记录。 | 重要 | 可按任务、用户、时间筛选并查看明细。 |

## 14. 非功能需求

| 序号 | 类别 | 需求说明 | 目标要求 |
|---|---|---|---|
| 1 | 性能 | 系统生成结果不能过慢，避免用户等待太久。 | 简单需求尽量在 30 秒内返回初版结果，复杂任务允许更长时间但需展示进度。 |
| 2 | 稳定性 | 系统在高并发或生成失败时仍能稳定工作。 | 任务失败可重试，不影响其他用户使用。 |
| 3 | 安全性 | 保护用户数据和下载内容安全。 | 支持登录鉴权、文件下载鉴权、输入内容过滤。 |
| 4 | 可扩展性 | 后续增加新硬件平台、新协议、新模板时改动成本低。 | 采用模板化、模块化设计。 |
| 5 | 兼容性 | 支持主流浏览器和不同分辨率设备。 | Chrome、Edge、Safari 等可正常访问。 |
| 6 | 可维护性 | 后台模板、器件库、代码规则应便于更新。 | 管理员可通过后台维护配置。 |
| 7 | 可用性 | 页面操作要简单直观，适合非专业用户。 | 用户能在少量步骤内完成“输入需求 - 生成 - 下载”。 |

## 15. 业务流程概述

1. 用户进入网页
2. 用户输入物联网项目需求
3. 系统将原始需求优化为提示词并展示给用户
4. 用户可直接采用优化后的提示词或手动修改
5. 系统根据优化后的提示词生成项目方案
6. 系统生成硬件电路图
7. 系统生成项目代码
8. 系统打包代码与说明文件
9. 用户在线预览并下载结果

## 17. 优化验收标准

| 验收项 | 测试方式 | 通过标准 |
|---|---|---|---|
| 结构化需求生成 | 输入一个完整需求，例如“使用 ESP32 + DHT22 做温湿度监测并用 OLED 显示”。 | 系统输出包含主控、传感器、联网方式、显示方式、供电方式的结构化结果。 |
| 提示词优化能力 | 输入一个不完整需求，例如“做一个智能家居系统”。 | 系统直接输出优化后的提示词，并标注原始输入、推荐值和可编辑项。 |
| 方案内容 | 查看生成的方案页。 | 方案中必须包含场景说明、器件清单、模块关系、原始输入和优化提示词。 |
| 电路图内容 | 打开电路图预览。 | 电路图必须显示具体主控和器件引脚连接，不能只显示抽象框图。 |
| 代码包完整性 | 下载 ZIP 并解压。 | ZIP 包内至少包含主程序、依赖说明、接线说明、运行步骤和 README。 |
| README 完整性 | 打开 README 文件。 | README 必须包含项目简介、硬件清单、接线、环境、运行步骤、注意事项。 |
| 自己项目访问 | 使用用户 A 账号访问用户 B 的项目链接。 | 系统拒绝访问，返回无权限提示或 403。 |
| 下载鉴权 | 未登录或使用过期下载链接尝试下载。 | 系统拒绝下载，并提示先登录或重新获取链接。 |
| 后台权限 | 使用普通用户账号访问后台管理页。 | 系统不显示后台入口或直接拒绝访问。 |
| 任务超时 | 模拟长时间生成任务。 | 任务应保持可追踪状态，用户可在历史记录中继续查看结果。 |

## 18. 风险与注意事项

- 自动生成的电路图和代码需要保证可读性和可落地性
- 对复杂需求可能会出现提示词优化偏差，因此需要允许用户手动编辑后再生成
- 不同硬件平台的代码模板可能存在差异，需要模板化处理
- 下载文件需要做好权限控制，避免非本人访问
- 生成逻辑应保留人工修改空间，避免完全黑盒化
- 物联网接线存在硬件差异，系统应明确提示用户核对实物和官方资料

## 19. 总结

本项目的核心价值在于把物联网项目从“文字需求”自动转换为“硬件方案 + 电路图 + 代码包”，大幅减少用户从想法到原型实现的时间成本。  
当前版本取消了多轮追问，改为 AI 提示词优化与推荐值策略，更适合快速生成和批量原型场景。  
如果作为第一版产品，建议先聚焦 STM32 + ESP32 以及常见传感器和执行器，确保生成结果稳定、可下载、可复用，再逐步扩展更复杂的硬件平台和高级能力。
